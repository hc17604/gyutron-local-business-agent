# GYUTRON Local Business Agent MVP 开发路线图

## 1. MVP 目标

在最短周期内交付一个可本地运行、可给真实客户演示、可用真实 Excel/CSV 数据生成老板日报的版本。

MVP 不追求平台 API 覆盖和复杂自动化，而是验证三件事：

1. 客户愿意把真实业务表格导入系统。
2. 系统能把混乱表格转成可分析的标准业务数据。
3. 老板日报能提供足够有价值的经营判断和行动建议。

## 2. 阶段划分

### Phase 0：项目骨架与基础约束

目标：把项目跑起来，明确本地数据边界。

交付物：

- React + Vite 前端骨架。
- FastAPI 后端骨架。
- SQLite 初始化。
- Docker Compose。
- `.env.example`。
- `data/uploads`、`data/reports` 本地目录。
- 基础健康检查接口。

验收标准：

- 本地开发模式前后端可启动。
- Docker Compose 可启动。
- `GET /api/v1/health` 返回正常。
- `data/` 不进入 Git。

### Phase 1：文件上传与预览

目标：支持用户上传 Excel/CSV，并预览表头和前 20 行。

交付物：

- 上传页面。
- 数据类型选择。
- 后端文件保存。
- pandas/openpyxl 解析。
- 上传记录入库。
- 解析失败错误提示。

验收标准：

- 可以上传 `.csv`、`.xlsx`。
- 可以选择询盘、订单、产品、库存、广告、客户、销售跟进。
- 页面显示表头和前 20 行。
- 文件保存在 `data/uploads`。
- 上传记录保存在 SQLite。

### Phase 2：字段自动猜测与手动映射

目标：让不同来源的表格字段映射到标准业务字段。

交付物：

- 标准字段字典。
- 字段检测规则。
- 映射编辑页面。
- 映射保存接口。
- 历史映射复用。

验收标准：

- 常见中英文表头能自动识别。
- 用户可以修改每个原始列对应的标准字段。
- 用户可以忽略无关列。
- 映射保存后刷新页面不丢失。
- 相似表头再次上传时优先复用历史映射。

### Phase 3：模型配置与 LLM Adapter

目标：用户可以自行配置 OpenAI-compatible 模型。

交付物：

- 模型设置页面。
- LLM 配置 CRUD。
- 测试连接接口。
- OpenAI-compatible adapter。
- 外部模型调用隐私提示。

验收标准：

- 用户可以保存 Base URL、API Key、模型名称。
- 用户可以设置默认模型。
- 测试连接成功或失败有明确反馈。
- 后端通过统一 adapter 调用模型。

### Phase 4：数据摘要与老板日报生成

目标：基于已映射数据生成老板日报。

交付物：

- 数据摘要服务。
- 老板日报 prompt。
- 报告生成接口。
- 报告状态记录。
- 报告保存到 SQLite。
- 报告详情页。

验收标准：

- 用户可以选择上传数据生成日报。
- 系统先生成结构化摘要，再调用 LLM。
- 报告包含经营结论、关键数字、异常提醒、重点客户、重点产品、市场趋势、跟进任务。
- 生成失败时保存错误信息。
- 报告可以在历史页面重新打开。

### Phase 5：业务规则

目标：用户通过自然语言规则影响日报分析重点。

交付物：

- 业务规则页面。
- 新增、编辑、启用、停用规则。
- 报告生成时注入启用规则。
- 报告保存规则快照。

验收标准：

- 用户可以新增规则，例如“超过 24 小时未回复的询盘要预警”。
- 停用规则后不再影响新报告。
- 报告详情能看到生成时使用的规则快照。
- 同一份数据在不同规则下生成的报告重点有明显变化。

### Phase 6：演示样例与打磨

目标：让销售演示和客户试用顺畅。

交付物：

- 脱敏样例 CSV。
- README 快速启动。
- 错误提示优化。
- 空状态和加载状态。
- 基础测试。

验收标准：

- 新机器按照 README 能在 10 分钟内跑起来。
- 使用样例数据能完整走通：上传、映射、配置模型、生成日报、查看历史。
- 至少覆盖文件解析、字段检测、报告生成核心测试。

## 5. MVP 风险与控制

### 5.1 字段混乱导致识别不准

控制方式：

- 自动猜测只是辅助，必须允许用户手动修正。
- 保存历史映射，减少重复配置。
- 优先覆盖常见中英文表头。

### 5.2 LLM 编造结论

控制方式：

- 后端先做确定性摘要。
- Prompt 明确要求不得编造。
- 报告中区分“已知事实”和“建议判断”。
- 对数据不足的地方输出“不足以判断”。

### 5.3 客户担心数据安全

控制方式：

- 文档和 UI 明确：数据默认保存在本地。
- 只有报告生成摘要会发给用户配置的模型服务商。
- 不接 GYUTRON 云端。
- 后续加入 API Key 加密和本地访问密码。

### 5.4 范围膨胀

控制方式：

- MVP 不接平台 API。
- MVP 不做自动化动作。
- MVP 不做复杂权限。
- 每个新需求都先判断是否服务于“本地老板日报”。

## 6. 演示路径

推荐销售演示流程：

1. 打开本地 Web 控制台。
2. 展示模型配置页面，说明客户使用自己的 API Key。
3. 上传订单样例 CSV。
4. 展示系统自动识别字段。
5. 手动修正一两个字段，说明可控。
6. 新增业务规则：“毛利低于 20% 的订单标红，并给出处理建议。”
7. 生成老板日报。
8. 展示报告中的关键结论、异常提醒、重点客户和跟进任务。
9. 打开历史报告，说明所有记录保存在本地。
