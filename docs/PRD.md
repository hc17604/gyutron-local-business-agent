# GYUTRON Local Business Agent 产品需求文档

## 1. 产品概述

**项目名称**：GYUTRON Local Business Agent

**产品定位**：面向跨境制造企业、跨境电商企业、工业品外贸公司的本地化 AI 经营 Agent。系统部署在客户本地电脑、办公室服务器或企业内网环境中，默认不把客户业务数据上传到 GYUTRON 服务器。

第一版 MVP 聚焦一个可以演示、可以真实试用、可以低风险交付的能力：**本地老板日报 Agent**。用户上传 Excel/CSV 数据，配置字段映射和业务规则，系统整理数据摘要并调用用户自行配置的 AI 模型生成老板日报，报告和规则保存在本地 SQLite 数据库中。

## 2. 产品原则

1. **本地优先**：客户业务数据、字段映射、历史报告、业务规则、上传文件默认保存在本地。
2. **客户自带模型密钥**：用户自行配置 OpenAI-compatible API Base URL、API Key 和模型名称，后续再扩展 Claude、Gemini、DeepSeek、Qwen、Ollama 等适配器。
3. **只读数据，低风险动作**：MVP 只读取上传文件、生成分析和报告，不修改平台数据，不自动发邮件，不自动执行订单、广告、库存、客户跟进等高风险动作。
4. **先文件导入，后平台 API**：MVP 不接 Alibaba、Amazon、Shopee、TikTok Shop、Shopify 等真实平台 API，先通过 Excel/CSV 验证业务价值。
5. **面向老板和业务负责人**：输出不是普通 BI 图表，而是可行动的经营日报、异常提醒、重点客户、产品表现、国家市场趋势和跟进建议。
6. **可解释、可修正、可沉淀**：字段映射、业务规则和历史报告都可查看、修改和复用，逐步形成企业本地记忆。

## 3. 目标用户

### 3.1 主要用户

- 外贸工厂老板、总经理、业务负责人。
- 跨境电商公司负责人、运营主管。
- 工业品贸易公司销售总监、跟单主管。

### 3.2 使用场景

- 每天导入昨天或最近一段时间的询盘、订单、广告、库存、客户跟进数据。
- 自动生成老板日报，快速了解经营重点和异常。
- 通过自然语言规则影响分析口径，例如：
  - “以后巴西客户优先级提高。”
  - “超过 24 小时未回复的询盘要预警。”
  - “毛利低于 20% 的订单标红。”
  - “重点看高客单价但低回复率的国家。”

## 4. MVP 范围

### 4.1 必须包含

1. 本地 Web 控制台。
2. Excel/CSV 文件上传。
3. 用户选择数据类型：
   - 询盘
   - 订单
   - 产品
   - 库存
   - 广告
   - 客户
   - 销售跟进
4. 系统读取表头和前 20 行样例数据。
5. 系统自动猜测字段含义。
6. 用户手动修正字段映射。
7. 字段映射规则保存到 SQLite。
8. 用户配置 AI 模型 API Key、Base URL、模型名称。
9. 系统将结构化数据整理成经营摘要。
10. 调用 LLM 生成老板日报。
11. 报告保存到本地 SQLite。
12. 用户查看历史报告。
13. 用户新增、编辑、启用、停用业务规则。
14. 生成报告时自动参考业务规则。

### 4.2 明确不做

MVP 不做以下能力：

- 不做 SaaS 多租户。
- 不默认上传客户数据到云端。
- 不接 Alibaba、Amazon、Shopee、TikTok Shop、Shopify 等平台 API。
- 不做复杂 RBAC 权限系统。
- 不自动发送邮件、企业微信、Slack 或 WhatsApp。
- 不自动修改平台数据、广告预算、订单状态、库存数据。
- 不做复杂工作流编排。
- 不做企业级审计、SSO、审批流。
- 不做本地大模型部署管理，只预留 Ollama 等本地模型接口。

## 5. 核心用户流程

### 5.1 首次配置

1. 用户打开本地 Web 控制台。
2. 进入模型配置页面。
3. 输入 Provider 名称、API Base URL、API Key、模型名称。
4. 点击测试连接。
5. 系统保存配置到本地 SQLite。

### 5.2 导入数据

1. 用户选择数据类型。
2. 上传 Excel 或 CSV 文件。
3. 后端保存原始文件到 `data/uploads`。
4. 系统解析表头和前 20 行样例。
5. 系统基于字段名称和样例值猜测业务字段。
6. 用户在页面上修正字段映射。
7. 系统保存字段映射和导入任务记录。

### 5.3 生成老板日报

1. 用户选择一个或多个已导入数据集。
2. 用户选择报告日期范围。
3. 系统加载启用中的业务规则。
4. 系统将数据按业务字段整理成摘要。
5. 系统调用 LLM 生成老板日报。
6. 系统保存报告标题、正文、摘要、使用的数据集、规则快照和模型信息。
7. 用户在历史报告页面查看报告。

### 5.4 维护业务规则

1. 用户打开业务规则页面。
2. 输入自然语言规则。
3. 选择规则适用的数据类型或全局适用。
4. 启用规则。
5. 后续生成报告时，系统把启用规则注入日报生成提示词。

## 6. 数据类型与标准字段

字段映射的目标不是建立复杂主数据系统，而是把不同表格的列统一到足够生成日报的标准字段。

### 6.1 询盘 Inquiry

建议标准字段：

- `inquiry_id`
- `created_at`
- `customer_name`
- `customer_email`
- `country`
- `source_platform`
- `product_name`
- `quantity`
- `message`
- `status`
- `owner`
- `last_reply_at`
- `reply_status`

关键分析：

- 新增询盘数量。
- 国家分布。
- 产品需求热度。
- 未回复或超时未回复询盘。
- 高价值客户线索。

### 6.2 订单 Order

建议标准字段：

- `order_id`
- `order_date`
- `customer_name`
- `country`
- `source_platform`
- `product_name`
- `quantity`
- `revenue`
- `cost`
- `gross_profit`
- `gross_margin`
- `currency`
- `status`
- `owner`

关键分析：

- 销售额、订单数、客单价。
- 国家市场表现。
- 产品销售表现。
- 低毛利订单。
- 大客户订单。

### 6.3 产品 Product

建议标准字段：

- `product_id`
- `sku`
- `product_name`
- `category`
- `price`
- `cost`
- `gross_margin`
- `stock_quantity`
- `status`

关键分析：

- 高询盘低成交产品。
- 高销售额产品。
- 高毛利产品。
- 低库存或滞销产品。

### 6.4 库存 Inventory

建议标准字段：

- `sku`
- `product_name`
- `warehouse`
- `stock_quantity`
- `reserved_quantity`
- `available_quantity`
- `reorder_level`
- `updated_at`

关键分析：

- 缺货风险。
- 安全库存预警。
- 高库存占用。

### 6.5 广告 Ads

建议标准字段：

- `date`
- `platform`
- `campaign_name`
- `product_name`
- `impressions`
- `clicks`
- `spend`
- `leads`
- `orders`
- `revenue`

关键分析：

- 花费、点击、询盘、订单。
- 高花费低转化广告。
- 高 ROI 活动。

### 6.6 客户 Customer

建议标准字段：

- `customer_id`
- `customer_name`
- `country`
- `email`
- `phone`
- `company`
- `source_platform`
- `customer_level`
- `last_contact_at`
- `owner`
- `notes`

关键分析：

- 重点客户。
- 长时间未跟进客户。
- 国家市场客户质量。

### 6.7 销售跟进 Sales Follow-up

建议标准字段：

- `followup_id`
- `customer_name`
- `country`
- `owner`
- `followup_at`
- `next_followup_at`
- `status`
- `content`
- `related_inquiry_id`
- `related_order_id`

关键分析：

- 今日应跟进客户。
- 超期未跟进客户。
- 高价值机会推进情况。

## 7. 字段映射需求

### 7.1 自动猜测

系统基于以下信息猜测字段：

- 表头名称。
- 表头中英文关键词。
- 前 20 行样例值。
- 数据类型，例如日期、数字、邮箱、国家、币种。
- 同一数据类型历史保存的字段映射。

### 7.2 手动修正

用户可以将原始列映射到标准字段，也可以选择忽略某列。

映射状态：

- `auto_mapped`：系统自动映射。
- `user_confirmed`：用户确认。
- `user_modified`：用户修改。
- `ignored`：用户忽略。

### 7.3 保存与复用

字段映射按数据类型、文件表头签名和用户修正结果保存。后续上传相似表格时优先复用历史映射。

## 8. 老板日报内容要求

日报必须面向决策，不只是描述数字。

建议结构：

1. **今日经营结论**：3-5 条最重要结论。
2. **关键数字**：询盘、订单、销售额、毛利、广告花费、库存风险等。
3. **异常提醒**：未回复询盘、低毛利订单、高花费低转化广告、库存风险。
4. **重点客户**：值得老板关注的客户和原因。
5. **重点产品**：高潜力、高风险、低毛利或高增长产品。
6. **国家/市场趋势**：按国家或区域总结机会与风险。
7. **销售跟进任务**：建议今天跟进谁、为什么、怎么跟进。
8. **需要老板拍板的问题**：价格、交期、重点市场、重点客户资源投入。

## 9. 本地记忆

MVP 的本地记忆包括：

- 模型配置。
- 上传文件记录。
- 字段映射。
- 业务规则。
- 历史报告。
- 报告生成时使用的数据集和规则快照。

后续版本可以扩展：

- 客户画像。
- 产品知识库。
- 市场偏好。
- 销售话术。
- 平台 API 同步记录。
- 工作流执行历史。

## 10. 非功能需求

### 10.1 隐私与数据边界

- 默认所有文件和数据库保存在本机。
- 不向 GYUTRON 服务器上传客户业务数据。
- 调用外部 LLM 时，需要在 UI 中明确提示：生成报告会把整理后的摘要发送到用户配置的模型服务商。
- API Key 存储在本地数据库。MVP 可先做基础存储，正式商用前需要加入加密存储或系统密钥环。

### 10.2 可部署性

- 支持 Docker Compose 一键启动。
- 支持本地开发模式分别启动前端和后端。
- 数据目录固定为 `data/`，便于备份和迁移。

### 10.3 稳定性

- 文件解析失败时给出明确错误。
- LLM 调用失败时保留数据摘要，不丢失导入记录。
- 报告生成任务需要记录状态：等待中、生成中、成功、失败。

### 10.4 性能边界

MVP 目标：

- 单文件建议不超过 50MB。
- 单次日报建议汇总 1-7 天数据。
- 前端只预览前 20 行。
- 后端生成报告前先做摘要，不把完整大表直接塞给 LLM。

## 11. 成功标准

MVP 成功的定义：

1. 可以在一台客户电脑或本地服务器跑起来。
2. 用户可以上传真实业务 Excel/CSV。
3. 系统能稳定识别表头、让用户修正字段映射。
4. 系统能生成一份老板愿意每天看的经营日报。
5. 业务规则能明显改变报告重点。
6. 历史报告和映射规则可以复用。
7. 演示时无需连接任何真实电商平台 API。
